## Motivation

Network callout drivers often need to defer a verdict on a connection
or packet while waiting for an asynchronous decision from another
component -- typically a user-mode policy service or a kernel-mode
classification driver. WFP provides several async mechanisms at
different layers (for example `FwpsPendOperation` /
`FwpsCompleteOperation` at ALE authorize layers), but eBPF programs
running through netebpfext have no way to express "pend this
operation and complete it later."

This proposal adds a **pend/complete primitive** to netebpfext so an
eBPF program can:

1. **PEND** a network operation -- absorb it while an external
   orchestrator makes a decision asynchronously.
2. Instruct netebpfext to **resume processing** when the orchestrator
   delivers an action via the complete map; the action determines
   exactly how netebpfext processes the **completion**.

## Requirements

1. An eBPF program attached to a supported hook must be able to
   pend the current operation in netebpfext.
2. An external orchestrator must be able to deliver an action to
   netebpfext asynchronously.
3. The WFP semantics around pend/complete must be fully encapsulated
   in the hook implementation (i.e., no knowledge or consumption of
   any WFP details must leak to the eBPF program or the
   orchestrator).
4. Initial scope is the `BPF_PROG_TYPE_CGROUP_SOCK_ADDR` program
   type, attach types `BPF_CGROUP_INET4_CONNECT`,
   `BPF_CGROUP_INET6_CONNECT`, `BPF_CGROUP_INET4_RECV_ACCEPT`, and
   `BPF_CGROUP_INET6_RECV_ACCEPT`. Pend/complete may be extended to
   additional program / attach types in the future. The general
   paradigm documented here is expected to work for future program
   types, though WFP implementation details within netebpfext may
   differ and will be handled at implementation time.
5. Multiple clients attached to the same hook must each be able to
   pend independently. The pend mechanism itself must not cause
   one client's pend to interfere with another's (shared state
   collisions, lost notifications, dropped completions). Normal
   chain semantics still apply -- an earlier client's terminating
   action can end the chain before a later client runs.

## Design overview

The proposal has two phases:

1. **PEND** -- the program calls a helper to defer its verdict,
   and netebpfext prepares the internal state needed to drive
   the eventual completion.
2. **COMPLETE** -- the orchestrator inserts an action into a
   program-declared map, and netebpfext resumes (or terminates)
   the operation accordingly. Initial action types:
    - `REJECT` -- final verdict = REJECT.
    - `PROCEED_SOFT` / `PROCEED_HARD` -- equivalent to a
      synchronous return of the same verdict from this slot.
    - `REINVOKE` -- re-invoke the pended program.

## PEND overview

**The `bpf_sock_addr_pend()` helper.** A program decides to pend by calling:

```
int bpf_sock_addr_pend(struct bpf_map *complete_map, pend_key_t *key, uint32_t flags);

typedef struct _pend_key {
    ebpf_extension_header_t header;  // standard versioning header
    uint64_t pend_id;                // globally unique among live
                                     // pends; generated by
                                     // netebpfext at pend time
} pend_key_t;
```

`complete_map` is a pointer to the program-declared complete map
that the orchestrator will later use to drive resumption. See
[COMPLETE overview](#complete-overview) for the map's declaration
and binding mechanics.

`key` is an in/out parameter. The caller MUST initialize the
`ebpf_extension_header_t` before calling; on success the helper
validates the header and fills in the freshly-generated pend id.
This key is the program's handle to the pended classify; the
program's only obligation is to forward it to its orchestrator
over whatever notification channel the consumer chose (BTF-resolved
function, ringbuf, etc.) so the orchestrator can later complete the
pend.

`pend_key_t` properties:
- **Unique per classify instance.** One key per chain invocation,
  shared across all clients that pend that same classify in the
  multi-attach case. The program and its user-mode orchestrator
  can therefore safely use it as a unique identifier for the
  pended connection (e.g., as a map key for per-pend metadata,
  or as a correlation token across components).
- **Opaque handle.** The program and orchestrator should treat it
  as a handle: use it to correlate the pend notification with the
  later complete, but do not interpret or rely on its contents.

`flags` is reserved for future use and MUST be 0. The helper
rejects non-zero values with a non-zero error return so that
future flags can be introduced without an ABI break.

The helper returns 0 on success and non-zero on failure.

> **REINVOKE re-entry.** When the pending client is
> re-invoked via a `REINVOKE` completion and chooses to pend
> again, it calls `bpf_sock_addr_pend()`. The helper detects
> (via the internal pend table) that the existing pend entry is
> still alive, reuses it, returns the **original** pend key to
> `*key`, and skips a second `FwpsPendOperation()` call.
> The helper transitions the entry's lifecycle back to `PENDED`
> and refreshes `insertion_time` so the stale-pend watchdog
> measures the new pend cycle. If the re-invoked program returns
> a terminal verdict instead, no `bpf_sock_addr_pend()` call is
> needed and the existing entry is processed normally.

> **Implicit program context.** The helper signature deliberately
> omits an explicit `ctx` parameter. netebpfext declares
> `bpf_sock_addr_pend` using the existing `implicit_context` extension
> feature (see [eBpfExtensions.md](eBpfExtensions.md), section
> *Note about `implicit_context`*): the program context is
> delivered as a hidden 6th argument set by the verifier/JIT glue,
> with dummy placeholders for the unused 2nd-5th arguments. No new
> platform plumbing is required, and the BPF-visible signature
> stays clean.

**New `PEND` verdict.** In addition to invoking `bpf_sock_addr_pend()`, the
program may return a new verdict `BPF_SOCK_ADDR_VERDICT_PEND`. When this verdict
is returned, it is a signal to netebpfext to freeze the chain at this slot and
wait for the complete map insertion. The program may choose to instead return any
other verdict, in which case, netebpfext does NOT wait for the completion and instead
handles the verdict inline. This provides an immediate response in cases when a complete
map insertion is not expected to occur (for example, when the notification mechanism
fails).

**Internal pend state.** When `bpf_sock_addr_pend()` runs, netebpfext records
an entry in a kernel-only hash table keyed by the pend key. The following
is the proposed structure of the state:

```c
typedef struct _wfp_completion_state {
    ebpf_attach_type_t layer_id;     // discriminator -- reuses the
                                     // existing attach-type enum
    ULONG classifyfn_cpu;            // CPU at pend time; recorded as
                                     // the target for the eventual WFP
                                     // completion callback
    union {
        // CONNECT (outbound) -- FwpsPendOperation / clone-reinject path
        struct {
            HANDLE completion_context;
            PNET_BUFFER_LIST cloned_nbl;
            UINT64 endpoint_handle;
            UINT8 remote_address[16];
            SCOPE_ID remote_scope_id;
            WSACMSGHDR* control_data;   // deep-copied into extension-
                                        // owned storage at pend time
            ULONG control_data_length;
        } connect;

        // RECV_ACCEPT (inbound) -- FwpsPendOperation / clone-reinject path
        struct {
            HANDLE completion_context;
            PNET_BUFFER_LIST cloned_nbl;
            IF_INDEX interface_index;
            IF_INDEX sub_interface_index;
            ULONG ip_header_size;
            ULONG transport_header_size;
            ULONG nbl_offset;
            BOOLEAN ip_sec_protected;
            UINT8 protocol;
            UINT8 local_address[16];
            UINT8 remote_address[16];
        } recv_accept;

        // future arms (for other WFP layer state)
    } u;
} wfp_completion_state_t;

// Lifecycle state -- gates ordering between the WFP-pend API call,
// notification, and the orchestrator's complete-map write. Lives on
// the current pender (see pender_info_t below) since each pender
// transitions independently.
typedef enum _pend_lifecycle {
    PEND_LIFECYCLE_PENDED = 0,             // entry installed and (on
                                           // success) WFP pend API
                                           // issued; awaiting an
                                           // orchestrator action
    PEND_LIFECYCLE_COMPLETION_PENDING = 1, // action received; verdict
                                           // work item queued. Duplicate
                                           // / late inserts against this
                                           // pend key are rejected. For
                                           // `REINVOKE`, this value is
                                           // held across the re-entry
                                           // (so concurrent dup completes
                                           // remain rejected); if the
                                           // re-invoked program calls
                                           // `bpf_sock_addr_pend()` again,
                                           // the helper transitions this
                                           // back to `PENDED` as part of
                                           // reusing the existing entry.
} pend_lifecycle_t;

// Per-pender state. Captures everything specific to the currently-
// outstanding pender on this classify. Replaced/refreshed each time
// a new client pends against this entry (e.g., after a PROCEED_*
// resumes the chain and a later client in the same chain pends).
// Only one pender is outstanding at a time; the chain freezes at
// the PEND verdict, so there is no need to track multiple
// simultaneous penders.
typedef struct _pender_info {
    // The attached hook client that issued bpf_sock_addr_pend()
    // (per-attach netebpfext object tracked in
    // filter_context->client_contexts[]). The pend entry holds a
    // rundown reference on this client (acquired via
    // net_ebpf_extension_hook_client_enter_rundown) so the client
    // -- and the program it wraps -- cannot detach while this
    // pender is outstanding. PROCEED_* -> resume at the next slot
    // in the chain after this client; REINVOKE -> re-enter this
    // client.
    net_ebpf_extension_hook_client_t* client_context;

    // Per-map extension context for the complete map this pender
    // passed to bpf_sock_addr_pend(). Used as the identity for
    // cross-client validation; see Complete-map binding.
    void* complete_map_context;

    // KeQueryInterruptTime() at this pender's install (100-ns
    // units). Watchdog compares against a system-wide TTL constant.
    // Reset on each re-pend (REINVOKE -> bpf_sock_addr_pend() reuse),
    // so the TTL applies per pend cycle rather than to the total
    // lifetime of the entry. Doubles as pend-age telemetry.
    uint64_t insertion_time;

    // Number of REINVOKE-driven re-entries serviced for this
    // pender so far. Bumped each time the REINVOKE work item
    // re-enters this client. Used to enforce the REINVOKE cap
    // (see Program safety).
    uint32_t reinvoke_count;

    // Lifecycle of this pender's outstanding pend (see above).
    pend_lifecycle_t lifecycle;
} pender_info_t;

typedef struct _pend_entry {
    // (1) Program context for re-invocation / helpers.
    // Captured snapshot of the per-attach-type context wrapper (e.g.,
    // net_ebpf_sock_addr_t) holding the program-visible bpf_* context
    // (family, compartment_id, 5-tuple, ...) plus netebpfext-private
    // bookkeeping (token info, hook_id, redirect context, ...). WFP's
    // classify parameters live on the classifyFn stack and disappear
    // once the callback returns, so any classify-time pointers (control
    // data, NBL contents, etc.) are deep-copied into extension-owned
    // storage before classifyFn unwinds. This lets re-entry give the
    // program the same helper-visible state it saw originally.
    void* captured_hook_ctx;

    // (2) WFP completion state (layer-tagged union).
    wfp_completion_state_t wfp_completion;

    // (3) Chain-aggregated verdict accumulated across prior chain
    // slots (priority semantics). Spans the whole classify, not a
    // single pender.
    uint32_t aggregate_verdict;

    // (4) Filter context owning the chain this entry belongs to.
    // A reference is held (via REFERENCE_FILTER_CONTEXT) for the
    // lifetime of the pend entry so the filter -- and its
    // client_contexts[] array -- cannot be torn down while a
    // PROCEED_* might still need to walk it to re-acquire rundown
    // on the remaining chain members. Released alongside the
    // pending client's rundown ref at terminal-verdict cleanup.
    net_ebpf_extension_wfp_filter_context_t* filter_context;

    // (5) Currently-outstanding pender. Refreshed in place when a
    // new pender pends against this entry; the previous pender's
    // state is no longer relevant by that point because its
    // complete cycle has already concluded.
    pender_info_t current_pender;
} pend_entry_t;
```

## COMPLETE overview

**The complete map.** Completion is delivered through a new
custom map type, `BPF_MAP_TYPE_SOCK_ADDR_PEND_COMPLETE`, registered by
netebpfext. The map MUST be **declared by the eBPF program**. The
map itself is purely an orchestrator <-> netebpfext channel --
programs neither read nor write it. Netebpfext registers a
`preprocess_map_update_element` provider callback on the map type,
so each `bpf_map_update_elem` synchronously dispatches into the
extension on the inserting thread.

Program-side declaration:

```c
struct {
    __uint(type, BPF_MAP_TYPE_SOCK_ADDR_PEND_COMPLETE);
    __type(key, pend_key_t);
    __type(value, complete_entry_t);
    __uint(max_entries, 1024);
} pend_complete_map SEC(".maps");
```

The complete map type, action enum, and value struct:

```c
// New custom map type registered by netebpfext.
#define BPF_MAP_TYPE_SOCK_ADDR_PEND_COMPLETE  /* next free id */

// Orchestrator-supplied action carried in each complete-map entry.
// Mirrors ebpf_sock_addr_verdict_t (the sync chain-of-programs return
// values) plus an async-only REINVOKE.
typedef enum _ebpf_pend_complete_action {
    // Stop processing the chain of programs. Final WFP verdict = BLOCK.
    // Same as a sync BPF_SOCK_ADDR_VERDICT_REJECT.
    EBPF_PEND_COMPLETE_REJECT = 0,
    // Resume processing the chain of programs after the pending
    // client. Same as a sync BPF_SOCK_ADDR_VERDICT_PROCEED_SOFT
    // from this slot.
    EBPF_PEND_COMPLETE_PROCEED_SOFT,
    // Resume processing the chain of programs after the pending
    // client. Same as a sync BPF_SOCK_ADDR_VERDICT_PROCEED_HARD
    // from this slot.
    EBPF_PEND_COMPLETE_PROCEED_HARD,
    // Re-invoke the pending client. No sync equivalent.
    EBPF_PEND_COMPLETE_REINVOKE,
} ebpf_pend_complete_action_t;

// Versioned value struct so the contract can grow without
// rev'ing the map type.
typedef struct _complete_entry {
    ebpf_extension_header_t header; // allows for versioning information
    // fields required to process the completion
    ebpf_pend_complete_action_t action;
} complete_entry_t;
```

**Complete-map binding.** Each pend is bound at install time to
the specific complete-map instance the program passed to
`bpf_sock_addr_pend()`. netebpfext resolves the `complete_map`
helper argument (an `ebpf_map_t*` handed to the helper by the
runtime) to its per-map extension context -- the same
`map_context` pointer netebpfext itself allocated in
`preprocess_map_create` for that map (see
[eBpfExtensions.md][ebpfext-map-context]) -- using the standard
`map_context_offset` mechanism. It stores that context pointer
in the internal pend entry alongside the rest of the per-pend
state, where it serves as the identity used later to validate
complete-map inserts.

When the orchestrator later inserts into a complete map, the
preprocess callback receives the per-map context for the map
carrying the insert. It compares (pointer equality) against the
context stored in the matching pend entry. A mismatch is
rejected with `EBPF_ACCESS_DENIED` and the pend entry is left
untouched. This prevents an unrelated eBPF client from driving
completion for a pend it did not originate, even if it somehow
learned the opaque pend key.

The orchestrator obtains its fd to the program-declared complete
map through standard discovery (looking it up by name on the
loaded program object, or via a pinned path) -- no new mechanism
is introduced.

[ebpfext-map-context]: ./eBpfExtensions.md#29-helper-functions-that-use-custom-maps

## Reference lifetimes across pend

Today, `net_ebpf_extension_hook_invoke_programs()` snapshots
`filter_context->client_contexts[]` under the filter lock, takes a
rundown reference on every client in the chain via
`net_ebpf_extension_hook_client_enter_rundown`, invokes them in turn,
then releases the rundown reference on **all** clients once the chain
finishes. This matches sync execution: every client in the chain is
held alive for the entire classify and released together when classify
returns.

Pend/complete cannot keep that same model. Once a bpf program
returns with a `PEND` verdict, the chain may be suspended for an
arbitrarily long time before a completion arrives -- holding a rundown
reference on every chain member across that gap would indefinitely
block detach on any of them. Instead:

- **At PEND time (inside `bpf_sock_addr_pend()`):**
  - The pending client's rundown reference is **transferred** from
    the classify-time array into `current_pender.client_context`.
  - When netebpfext processes the `PEND` verdict from the bpf program,
    It releases rundown on every **other** client in the
    snapshot (the normal end-of-classify release) but skips the
    pending client.
  - A reference is taken on the filter context via
    `REFERENCE_FILTER_CONTEXT` and stored in `pend_entry.filter_context`.
    This keeps the filter -- and the live `client_contexts[]`
    array -- valid for the duration of the pend, so a later
    `PROCEED_*` can safely walk it to re-acquire rundown.
- **When processing the `PROCEED_*` verdict (work item):**
  - Walk `pend_entry.filter_context->client_contexts[]` under
    the filter lock and snapshot it (same pattern as
    `net_ebpf_extension_hook_invoke_programs()`). The full walk
    is needed to **locate** the pending client
    (via `current_pender.client_context`); the array contents
    may have changed during the pend.
  - Call `net_ebpf_extension_hook_client_enter_rundown()` only
    on clients **after** the pending client in the snapshot --
    these are the clients we will invoke. Clients before the
    pending one already executed during the original classify
    and contributed to `aggregate_verdict`; they are not
    re-invoked and do not need rundown. The pending client's
    own rundown ref is already held by the entry from PEND time.
    Detached clients (no longer present in the array) are simply
    skipped.
  - Resume invocation at the slot **after** the pending client.
    If the resumed chain itself hits another `PEND` from a
    different client, the **At PEND time** logic above applies
    again to the new pending client (the snapshot refs held for
    the previous pender plus the now-irrelevant slots after the
    new pending one are released as "all other" refs). If no
    further client pends, the **terminal-verdict** logic below
    applies.
- **When processing the `REINVOKE` verdict (work item):**
  - The pending client's rundown reference is already held by
    the entry, so no further lifetime management is needed. The
    rundown reference remains held until the `COMPLETE` is issued.
- **Processing a terminal verdict:**
  - Release rundown on **all** clients with held refs -- the
    pending client (`current_pender.client_context`) plus any
    clients after the pending one in the most recent `PROCEED_*`
    snapshot whose rundown is still held.
  - `DEREFERENCE_FILTER_CONTEXT(pend_entry.filter_context)`.
  - Remove the pend entry.

If the pending client itself detaches while a pend is outstanding,
detach blocks on rundown drain until the pend completes (the entry
holds the only ref keeping it from running down). This is the same
shape the sync chain already relies on -- detach cannot tear down
a client whose code is still active on some thread.

## Walkthroughs

### PEND walkthrough

1. **Program decides to pend.**
   - The program is invoked in the chain and determines that
     the operation needs an async decision.
   - It calls `bpf_sock_addr_pend(&pend_complete_map, &key, 0)`.
2. **netebpfext processes the helper.**
   - **Resolves the `complete_map` argument** to its per-map
     extension context via the standard `map_context_offset`
     mechanism.
   - **Generates the pend key** and prepares to write it to
     `*key`.
   - **Captures the running chain aggregate** (from prior
     slots in the current classify invocation) into the
     entry's `aggregate_verdict`.
   - **Builds the internal pend entry** (resolved map context,
     captured hook context, etc.) and inserts it into the
     table **before** issuing the WFP pend API -- so any
     synchronous callback from inside that WFP API can find
     the entry.
   - **Synchronously issues the layer-appropriate WFP pend API.**
   - **On success:** returns 0 with the pend key written to
     `*key`. The classify wrapper transfers the pending
     client's rundown ref and takes a filter-context ref per
     the **At PEND time** rules in
     [Reference lifetimes across pend](#reference-lifetimes-across-pend).
   - **On WFP failure:** removes the just-inserted pend entry
     and returns non-zero. The rundown reference was never
     transferred and is released normally when classify
     returns. The program then falls through to a normal
     synchronous verdict.
3. **Program notifies the orchestrator.**
   - After a successful `bpf_sock_addr_pend()`, the program sends its
     notification (BTF-resolved function, ringbuf, etc.) so
     the orchestrator can later complete the pend.
4. **Program returns its verdict.**
   - **`BPF_SOCK_ADDR_VERDICT_PEND`:** signals netebpfext to
     freeze the chain at this slot. Subsequent programs are
     not invoked for this classify until the orchestrator
     drives a resumption via the complete map. Continue to
     step 5.
   - **Any other verdict:** netebpfext does **not** wait for a
     complete-map insertion and handles the verdict inline
     using normal sync chain semantics. The dangling pend
     entry is cleaned up synchronously by the classify wrapper
     (it synthesizes a `REJECT`-equivalent WFP completion and
     removes the entry). The program uses this path when it
     wants to short-circuit the async wait -- for example,
     when the notification mechanism failed (see
     [Notification fails after a successful pend](#notification-fails-after-a-successful-pend)).
5. **Classify is suspended pending completion.**
   - The classify wrapper returns the layer-appropriate
     "operation pended" action to WFP (for example, the
     absorb-style flags used at the ALE layers).
   - WFP holds the calling thread/operation in the pended
     state.
   - The pend entry now lives in the table awaiting a complete map
     update.

### COMPLETE walkthrough

1. **Orchestrator inserts into the complete map.**
   - The orchestrator calls
     `bpf_map_update_elem(complete_map_fd, &key, &value, flags)`
     with the pend key from the notification and a
     `complete_entry_t` carrying the chosen action.
2. **netebpfext's `preprocess_map_update_element` callback runs.**
   - Looks up the matching internal pend entry by `key` and
     validates the insert. Three rejection cases, each returned
     synchronously to the orchestrator's `bpf_map_update_elem`
     call with no side effect:
       - **No entry for `key`** -- returns `EBPF_KEY_NOT_FOUND`.
         Either the key is bogus, or the watchdog already
         force-completed this pend
         (see [Stale-pend watchdog](#stale-pend-watchdog)).
       - **`map_context` mismatch** -- the per-map context the
         callback received does not equal the one stored in the
         entry. Returns `EBPF_ACCESS_DENIED`. See
         [Cross-client complete attempt](#cross-client-complete-attempt).
       - **Entry already in `COMPLETION_PENDING`** -- a complete
         for this key is already being processed. Returns
         `EBPF_OBJECT_ALREADY_EXISTS`. See
         [Duplicate completes](#duplicate-completes).
   - **On a valid insert:** copies the action and any needed
     state onto a work-item context, transitions the entry's
     lifecycle to `COMPLETION_PENDING`, queues a work item to
     process the action, and returns success. Re-entry cannot
     run inline -- the program may immediately re-enter map
     CRUD which would deadlock.
3. **Work item executes.**
   - Removes the complete-map entry first, then processes the
     recorded action. Removing the complete-map entry first
     frees the slot so a `REINVOKE` followed by a re-pend
     (which symmetrically reuses the **same** pend key) can
     land a new entry without a duplicate-key collision. The
     pend entry is **not** removed here.
   - Per-action behavior:
       - **`REJECT`:** terminal -- proceeds to step 4 with a
         BLOCK verdict. No further programs in the chain are
         invoked.
       - **`PROCEED_SOFT` / `PROCEED_HARD`:** resumes invocation
         at the slot **after** the pending client. Rundown
         re-acquisition is described in
         [Reference lifetimes across pend](#reference-lifetimes-across-pend).
         Verdicts from resumed slots fold into the entry's
         `aggregate_verdict` under the standard "most restrictive
         wins" rule. On a terminal verdict (or chain end), proceeds
         to step 4. If a different client in the resumed chain itself
         pends, the entry transitions back to PEND state (new
         `current_pender`).
       - **`REINVOKE`:** re-enters the pending client directly
         (its rundown ref is already held by the entry; no
         filter-context walk needed). No lifecycle reset is
         performed before re-entry -- if the re-invoked program
         calls `bpf_sock_addr_pend()` again, the helper
         transitions `COMPLETION_PENDING` -> `PENDED` as part of
         reusing the existing entry (see
         [`bpf_sock_addr_pend()` REINVOKE re-entry](#pend-overview)).
         The re-entry yields either a terminal verdict (proceeds
         to step 4) or a new pend (returns to the
         [PEND walkthrough](#pend-walkthrough)).
4. **Terminal-verdict cleanup.**
   - Once a terminal verdict is reached (via any path in
     step 3), netebpfext drives the layer-specific WFP
     completion API, releases rundown on **all** clients with
     held refs (the pending client plus any snapshot clients
     still held from processing the last `PROCEED_*`), drops the
     filter-context reference (`DEREFERENCE_FILTER_CONTEXT`),
     and removes the internal pend entry.
   - Per-pend completion occurs exactly-once.

## Program safety

A pended operation consumes kernel resources until it is completed.
Three backstops bound the impact pend consumers can have on the system
if they pend without bound, fail to drive completions, or ping-pong
indefinitely between PEND and REINVOKE:

### Systemwide pend memory quota

netebpfext enforces a maximum amount of memory that may be held by
simultaneously outstanding pend entries across the entire system.
The per-entry cost is layer- and operation-dependent (for example,
inbound packet pends include cloned NBL payload whose size varies
per operation), so the quota is expressed in bytes rather than as
a fixed entry count. When a `bpf_sock_addr_pend()` call would push the
global outstanding pend memory over the limit, the helper returns
`-ENOMEM` without creating any internal pend state or invoking any
WFP pend APIs. The program MUST handle this error appropriately.

### Stale-pend watchdog

netebpfext periodically walks the internal pend table and force-
completes any entry whose age (`now - insertion_time`) exceeds a
maximum timeout, issuing a `REJECT` verdict on the program's
behalf. `insertion_time` is reset on each re-pend cycle (see
[PEND overview](#pend-overview)), so the TTL applies per pend
cycle rather than to the total lifetime of the entry. This bounds
the lifetime of any individual pend cycle regardless
of orchestrator behaviour.

If the orchestrator does eventually issue a complete for a pend
that has already been force-completed by the watchdog, the
complete-map insert finds no matching pend entry and is rejected
with `EBPF_KEY_NOT_FOUND` (see
[COMPLETE overview](#complete-overview)).

### REINVOKE iteration cap

A program that is re-invoked via `REINVOKE` is allowed to call
`bpf_sock_addr_pend()` again, which reuses the existing pend entry
(see [PEND overview](#pend-overview)). Without a bound, a
misbehaving program / orchestrator pair could ping-pong indefinitely
between PEND and REINVOKE on the same operation, holding kernel
resources and starving the underlying connection.

netebpfext bounds this by tracking `reinvoke_count` on the pend
entry: every REINVOKE work item bumps the counter before re-entering
the program. When `bpf_sock_addr_pend()` is called on a re-invoked
entry whose `reinvoke_count` has reached the system-wide cap, the
helper returns a non-zero error code without re-issuing the WFP
pend API. The program is expected to handle this error and return
a synchronous (non-PEND) verdict; if it nonetheless attempts to
return `BPF_SOCK_ADDR_VERDICT_PEND` after the helper failed,
classify-exit cleanup synthesizes a `REJECT` (same handling as the
[notification-fails edge case](#notification-fails-after-a-successful-pend)).

The cap value is a netebpfext internal constant chosen to allow
realistic async chains while well below any DoS-relevant depth.

## Edge cases

The pend/complete mechanism has a few notable edge cases that
need explicit handling. In all cases below, the listed error code
is returned by netebpfext from its `preprocess_map_update_element`
callback and propagated synchronously by the eBPF runtime to the
orchestrator's `bpf_map_update_elem` call (no entry is inserted
into the complete map for any of these rejections).

### Notification fails after a successful pend

`bpf_sock_addr_pend()` succeeded (WFP operation is pended, internal pend
entry exists) but the program's notification call to its orchestrator
returned failure. The program MUST return a non-PEND verdict to surface the
failure. netebpfext's classify wrapper detects the dangling pend
entry attached to this classify and synthesizes the same handling
the COMPLETE walkthrough would run for `REJECT`: drives the
layer-specific WFP completion and removes the internal pend entry.
The orchestrator never sees this pend.

### Duplicate completes

Nothing prevents an orchestrator from inserting the same pend
key into the complete map twice (retry, duplicate notification).
The second insert finds the pend entry already in
`COMPLETION_PENDING` state (or already removed) and is rejected
with `EBPF_OBJECT_ALREADY_EXISTS` (or `EBPF_KEY_NOT_FOUND` if
already removed); WFP completion runs exactly once per pend.

### Orchestrator completes before program returns from classifyFn

The orchestrator can race the program: receive the notification,
call `bpf_map_update_elem` on the complete map, and have the
preprocess callback run all before the program returns its `PEND`
verdict from classifyFn. This is benign by design:

- `bpf_sock_addr_pend()` issues the WFP pend API synchronously
  before returning, so WFP already knows the operation is pended
  by the time the program calls into the notification helper.
- The preprocess callback acquires the per-entry lock to validate
  and transition `PENDED -> COMPLETION_PENDING`, then queues the
  verdict work item; no inline re-entry into the program is
  possible.
- The work item runs on a separate thread at `PASSIVE`.
  For `REJECT` / `PROCEED_*` it calls the WFP completion API
  per the WFP contract; WFP's own serialization handles the case
  where completion is called while classifyFn is still on the
  stack of another thread.
- For `REINVOKE` the work item re-enters the program with the
  entry's lifecycle still at `COMPLETION_PENDING` (no eager
  reset). The original classify thread's return is independent
  of this re-entry; both invocations operate against the same
  pend entry through the entry lock, so neither sees inconsistent
  state. If the re-invoked program calls `bpf_sock_addr_pend()`,
  the helper transitions `COMPLETION_PENDING` -> `PENDED` as part
  of reusing the existing entry.

The original classify thread always returns its `PEND` verdict
normally; netebpfext's classify wrapper returns the
layer-appropriate "absorbed" action to WFP. Nothing on that path
depends on the pend entry's current lifecycle.

### Cross-client complete attempt

If an unrelated eBPF client somehow learns a pend key (leaked
through telemetry, log, replay, etc.) and tries to drive
completion through *its own* `BPF_MAP_TYPE_SOCK_ADDR_PEND_COMPLETE`
instance, the preprocess callback finds a pend entry for the
key but the `map_context` it receives does not match the one
stored in the entry. The insert is rejected with
`EBPF_ACCESS_DENIED`, the pend entry is left in `PENDED`
state, and only the original program's bound complete map can
drive resumption. This bounds the blast radius of a leaked
pend key to denial-of-correlation: the attacker cannot drive a
completion verdict for someone else's pend.

## Async orchestrator integration guide

The pend/complete mechanism is **orchestrator-agnostic** -- it
provides WFP lifecycle management, the pend key, the action
vocabulary, and the complete map. The orchestrator owns
everything else.

### Roles

| Component        | Role                                                                           |
|------------------|--------------------------------------------------------------------------------|
| eBPF program     | Declares the complete map, calls `bpf_sock_addr_pend()` with it, dispatches notification, returns `PEND` verdict. |
| Decision-maker   | Receives notification, decides, picks the action.                              |
| Complete-map writer | Holds an fd to the program-declared complete map; inserts `{key, action}` to drive resume. |

The eBPF program runs in-kernel; the decision-maker and
complete-map writer can be the same user-mode process, separate
processes, or a kernel driver.

### Notification mechanism (consumer's choice)

Notification is synchronous from the program's perspective so
failure can be surfaced inline (program falls back to a non-PEND
verdict). Common choices:

| Mechanism            | How it works                                                                                                | Trade-offs                                                                          |
|----------------------|-------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------|
| BTF-resolved function | Orchestrator driver exposes a BTF-resolved function; program calls it with the pend key + per-pend data.    | Synchronous, low-latency. Requires a kernel driver.                                 |
| Ringbuf / perf event | Program writes pend key + data to a consumer-owned ringbuf; orchestrator consumes from user mode.           | No kernel driver. Higher latency; ringbuf-full forces a non-PEND fallback.          |

Per-pend application data (rule context, connection metadata)
lives in consumer-owned regular BPF maps -- not in the
netebpfext-internal pend table or the complete map.
